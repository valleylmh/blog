# 我把一个AI换装Demo，慢慢做成了SASS产品

![OutfitLens 公众号封面](/images/articles/20260622/cover.png)

一开始就是一个点子，用在线编程平台Lovable开发了”一键换装“的网站，从需求到任务到开发，全程不碰代码实现，后面用gitHub做仓库管理，克隆到本地上继续开发，用已经订阅了的GPT-5.5重新开发升级了，之前是找AI Coding的感觉，现在是完全接受了AI Agent 开发。

从一键换装的网站，做着做着变成图片工具网站，主题变为”一张照片，重塑万千形象“。

> 上线地址：https://outfit.valleylmh.vip

![OutfitLens](/images/articles/20260622/index.webp)

一开始做 OutfitLens 的时候，我脑子里的画面很简单：用户上传一张人像，选择一个风格，然后生成一张新的照片。

这个想法听起来很像一个周末 demo。前端放一个上传框，后端接一个图片模型，结果回来以后展示出来。只要第一张图生成成功，就很容易让人产生一种错觉：产品好像已经差不多了。

真正做下去之后，我才发现，AI 图片生成最难的部分并不在“能不能生成”。更难的是，当生成这件事变慢、变贵、变不稳定之后，产品还能不能让用户安心地用下去。

用户看到的是一个按钮，但按钮后面其实有一整条链路。

图片要先压缩，否则一张原图就可能把请求和数据库撑得很难受；匿名用户要做人机验证；登录用户要检查积分，积分不够要提醒，扣了积分还要有流水；提示词要先过内容审核，不能等图片生成完了才发现不合规；任务不能一直卡在前端等待，得先入队，再由后台慢慢处理，最后把结果写回来。

也就是说，用户只是点了一次“生成”，产品却要同时处理图片、模型、队列、积分、支付、审核和失败状态。

我踩的第一个坑，是把生成当成普通接口。

普通接口最好是几百毫秒到几秒内返回。但图片生成不是这样，尤其是一些质量更高、场景更复杂的模型，等待两三分钟并不稀奇。如果前端一直等一个同步响应，用户会很快陷入一种尴尬状态：不知道它是在生成，还是已经坏了。

后来我把生成拆成了任务。用户提交之后，后端先创建 `image_jobs`，再由 worker 根据模型去处理。前端只轮询任务状态，告诉用户现在还在生成、失败了，还是已经完成。这个改动看起来偏工程，但它改变的是用户的心理感受：等待从一片黑箱，变成了一个能被理解的过程。

第二个坑，是图片传输。

开发早期用 base64 很方便，拿到图片就直接塞进请求里。但图片一大，问题就来了：请求体变大，数据库字段变重，webhook payload 也会变得不可靠。后来改成先把图片落到 Storage，再把 signed URL 写进任务里。这个细节用户看不见，但如果不做，后面所有稳定性都会被它拖住。

第三个坑，是积分。

我以为积分就是一个余额。后来发现不是。真正上线后，自认为用户关心的不是“还有多少积分”，而是“为什么少了这些积分”“这次失败有没有扣”“我买的积分为什么还没到账”。

所以积分必须变成流水。订阅发放、积分包购买、生成消耗、异常退款、后台调整，都要能追踪。否则支付一接进来，所有问题都会变成客服问题，而且很难解释清楚。

模板体系也是做着做着才想明白的。

职业照、证件照、毕业照、写真大片，这些是明确用途。用户点进去，是为了完成一个具体任务。但“我与自己”“我与人”“我与地方”“我与万物”更像进阶玩法，它们背后涉及多主体、多参考图、强场景合成和更复杂的提示词理解。

如果把它们全都混在一个模板列表里，产品看起来会更热闹，但用户会更困惑。基础模板应该服务快速生成，进阶玩法应该服务灵感和探索。这个边界不划清楚，最后就会变成什么都能点，但用户不知道哪个能力是真正稳定可用的。

还有一个很现实的坑，是首页。

做 AI 产品很容易想解释自己：用了什么模型、支持什么场景、为什么安全、为什么高清、为什么值得用。写着写着，首页就会从产品入口变成说明书。

但 OutfitLens 真正应该优先展示的，是可用的核心功能。用户来到这里，不是为了读完一篇产品介绍，而是想上传照片，看自己能变成什么样。能由图片、标题和点击行为自解释的地方，就不要再堆副文案。产品不是讲清楚才成立，而是跑起来才成立。

现在回头看，OutfitLens 已经不只是一个“能出图”的 demo。它有基础生成，有模板图鉴，有进阶玩法，有多模型选择，有异步队列，有积分和支付，有历史记录，也有内容审核和使用政策。

但这些功能并不是为了显得复杂。它们都是被真实问题逼出来的。

生成慢，所以需要队列。  
模型贵，所以需要积分。  
支付会延迟，所以需要回跳确认。  
微信验证不稳定，所以需要降级策略。  
图片太大，所以需要存储中转。  
玩法太多，所以需要信息架构。  
内容有风险，所以需要审核前置。

做完这些之后，我对 AI 产品有了一个更朴素的理解：模型能力只是起点，产品化才是长期工作。

把一个能力放到页面上不难，难的是让用户在失败、等待、付费、重试、下载、回看这些普通环节里，都不觉得慌。一个 AI 工具真正变成产品，不是因为它能生成一张惊艳的图，而是因为它能稳定地陪用户完成一次又一次具体的创作。

这也是 OutfitLens 这段时间给我的最大提醒：不要急着把想象力讲得很满，先把每一次点击后的体验做扎实。

AI 图片生成，不只是接个接口。它更像是把一团不确定的能力，慢慢整理成一个用户愿意相信的产品。

## 附录：功能清单

- 首页生成工作台：上传人像后直接选择换装、职业照、毕业照、证件照、未来的你、老照片修复。
- 模板图鉴：按职业形象、证件头像、毕业纪念、写真大片、国风 / 复古、创意未来组织模板。
- 进阶玩法：分为“我与自己”“我与人”“我与地方”“我与万物”四个方向。
- 多模型选择：OutfitLens、Seedream、Nano Banana、GPT Image，不同模型对应不同效果和积分成本。
- 异步生成队列：通过 `image_jobs` 入队，worker 后台处理，前端轮询状态。
- 并发限制：前端控制同时 processing 的任务数量，避免用户连续提交造成体验混乱。
- 图片处理：上传前压缩，后端通过 Storage signed URL 中转，避免大 base64 压垮任务链路。
- 积分体系：登录用户按模型扣积分，保留积分流水。
- 支付体系：支持订阅、积分包、支付回跳确认和延迟到账提示。
- 免费体验：匿名用户有每日限制，并结合人机验证和微信环境降级。
- 内容审核：生成前接入 Creem prompt moderation，审核失败不入队、不扣积分。
- 历史记录：登录用户可以查看、下载和复用历史结果。
- 双语文案：中英文覆盖导航、生成、支付、积分、错误提示和法务页面。
- 法务页面：隐私政策、服务条款、可接受使用政策。
